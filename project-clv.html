Customer Lifetime Value Analysis in E-Commerce

The Problem

A fast-growing e-commerce business had been relying on third-party tools to understand user behaviour. As the company scaled, those tools were no longer sufficient. The core business question was concrete: was the $2 per customer marketing acquisition cost actually justified? Without a rigorous CLV framework, there was no way to answer that with confidence — only instinct.

The Approach

Working with a parsed events table (raw_events) containing front-end user actions from November 2020 to January 2021, I filtered purchase events and used user_pseudo_id as the unique customer identifier. I built a BigQuery SQL framework using CTEs and window functions to calculate a sequential chain of metrics:

Average Order Value (AOV) → Purchase Frequency (PF) → Customer Value → Customer Lifetime Value (CLV), where CLV was derived as Customer Value multiplied by a 3-year assumed lifespan — documented explicitly as a parameter, not hardcoded, so the business could update it independently.

Key Findings

The average CLV was $2,950.43 — confirming the $2 acquisition cost was not just justified but represented an extraordinary return on investment of over 1,400×.

Purchase frequency analysis revealed which customer segments warranted higher re-engagement spend, separating high-value loyalists from one-time buyers with meaningfully different long-term profiles.

AOV and frequency together enabled value-based segmentation that third-party summaries could not provide — giving the marketing team a defensible, reproducible framework for budget allocation.

Impact

The analysis gave the marketing team a quantified, reproducible framework for acquisition spend decisions. The CLV calculation was also set up as a reusable SQL model so the business could run it against updated data each quarter without rebuilding the logic from scratch.

What I Learned

The most important step in CLV analysis is agreeing on the definition of 'lifetime' before writing any SQL. Using different assumptions changes the output dramatically. For this project I used 3 years, but I would always make that a documented parameter rather than a hardcoded value in future work — so the business can own and update the assumption rather than inheriting mine.
